  • Migrate Enterprise from SVN to Git

    August 2013 – May 2014

    I have been tasked with converting the University from SVN to Git. Our organization has roughly 16GB with 115k files across a few SVN repositories. I have been tasked from moving the entire organization from SVN to git. Throughout this process I have been installing and using GitLab for self hosted Git collaboration. We needed to be able to mirror vendor repositories as part of our development process and since GitLab doesn't support repository mirroring I wrote it myself (see gitlab-mirrors on Github). I wrote wrapper scripts to compliment gitlab-mirrors which auto-add mirrors as a vendor makes them available with no intervention.

    Not only am I in charge of coordinating maintence of the SVN repositories to ease transition from SVN to Git but I'm also single handedly creating all of the training materials and designed a curriculum bootcamp. It is for training Drexel developers on best practices such as the forking workflow and gitflow. The training materials include videos made using Camtasia studio in addition to written documentation. I keep it all in a git repository which I expose to the developers. This method is preferred because current employees can reference material and new employees can be trained using the same material.

    I have written scripts which dump all committers from SVN and generate an authors file by pulling in name and email from LDAP. This helps to facilitate proper naming and authorship in the user base. As part of this process I have contributed back to Gitlab (not by just the gitlab-mirrors project) by completely rewriting their Apache vhost config and Apache documentation. I will soon rewrite a lot of the documentation in the gitlab-recipes repository to reflect best practices for several configurations. If you would like to talk with me about this I hang out in the #gitlab channel on freenode as user sag47. I have screen session open 24x7 so if I don't immediately reply allow some time for asynchronous communication.

  • Automate Development Pipeline

    November 2012 – May 2014

    Drexel has been using a manual build and deploy process for in-house applications. I devised a completely automated deployment pipeline for testing.

    Using Jenkins, I set up job profiles that allow each developer to run a deployment which will automatically pull from SVN and deploy to the application server. The developers can even select multiple application deployments from the same job with minimal effort. I wrote in several safety checks which will abort deployments if developers have selected incompatible options. The error output is meaningful telling them where they went wrong and recommendations for how to fix it.

    The project is secured with a firewall to specific IP ranges, SSL, and authenticates against the University LDAP for ease of user management. The project is used for both the deployment of Java applications and to build .NET applications. The environment comprises both Windows and Linux nodes.

    Team members:
    • Sam Gleske

  • Helpdesk Chat Service (IRT TechChat)

    August 2008 – January 2009

    As a Helpdesk employee I found that the organization was missing a key component of communication with customers. Drexel did not have a web based help chat system.

    I proceeded to analyze available applications and decided to go with Openfire by IgniteRealtime. I built an Openfire server at home and became acquainted with the array of available features. When I felt proficient enough in the application I proposed it to Drexel University. First to my direct superior and then further up the chain. I created a powerpoint presentation, information packet, and even a live demo of the application. It involved a process of multiple presentations with general managers and eventually the Associate VP and CTO.

    During the project testing phase I assisted Core Technology Infrastructure (Drexel CTI) with configuring the Linux server, setting up security, and best practices. I also patched a few bugs in the software and tightened the security by excluding weak SSL/TLS ciphers for chat clients (compiled with Ant). I also assisted Information Technology Services (Drexel ITS) with customizing the template of the service to conform to the website design. This also included setting up multiple chatrooms for different areas of the University and template responses for common problems. Upon the production phase of the project I committed all bug fixes to the upstream Openfire project in the form of bug reports and patch submissions.

    I inherited administration responsibilities of the application when hired by Drexel CTI in September of 2011.
